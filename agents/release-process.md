# Release Process

## Creating a Release Candidate (RC) from dev branch

Release candidates allow testing pre-release versions from the dev branch before merging to main.

1. **Update pyproject.toml version with RC suffix:**
   ```bash
   # In pyproject.toml, set version to something like:
   version = "0.6.0rc0"  # Note: no dash, per PEP 440
   ```

2. **Commit version changes:**
   ```bash
   git add pyproject.toml
   git commit -m "Bump version to 0.6.0rc0"
   ```

3. **Create and push the git tag (with dash for git):**
   ```bash
   git tag v0.6.0-rc0  # Git tag uses dash
   git push origin v0.6.0-rc0
   git push origin dev
   ```

4. **GitHub Actions will automatically:**
   - Run tests
   - Verify tag is on dev or main branch (RC tags allowed on dev)
   - Generate `cicada/_version_hash.py` with the current tag and commit
   - Build the package
   - Publish to PyPI as a pre-release

5. **Test the RC installation:**
   ```bash
   uv tool install --pre cicada-mcp  # --pre flag to install pre-releases
   cicada --version  # Should show version and commit hash
   ```

6. **Iterate as needed:**
   - Make fixes on dev branch
   - Increment RC number: v0.6.0-rc1, v0.6.0-rc2, etc.
   - Repeat steps 1-5

## Creating a Stable Release from main branch

Stable releases must be tagged from the main branch.

1. **Merge dev to main:**
   ```bash
   git checkout main
   git merge dev
   git push origin main
   ```

2. **Update pyproject.toml version (remove RC suffix):**
   ```bash
   # In pyproject.toml, set final version:
   version = "0.6.0"
   ```

3. **Commit version changes:**
   ```bash
   git add pyproject.toml
   git commit -m "Bump version to 0.6.0"
   ```

4. **Create and push the git tag:**
   ```bash
   git tag v0.6.0
   git push origin v0.6.0
   git push origin main
   ```

5. **GitHub Actions will automatically:**
   - Run tests
   - Verify tag is on main branch (stable releases require main)
   - Generate `cicada/_version_hash.py` with the current tag and commit
   - Build the package
   - Publish to PyPI

6. **Test the installation:**
   ```bash
   uv tool install cicada-mcp
   cicada --version  # Should show version and commit hash
   ```

## Version Management

- Version number is stored in `pyproject.toml`
- Git tag and commit hash are stored in `cicada/_version_hash.py` (auto-generated, not committed)
- `cicada --version` or `cicada -v` displays version, tag, and commit hash
- Version format: `cicada 0.2.0 (v0.2.0-rc0/5ea1134)` - tag/hash format allows tracking RC releases
- **For PyPI installs:** Shows the version, tag, and hash from when the package was built by CI/CD
- **For local development:** The file is generated by `make pre-commit` and falls back to `git describe --tags` and `git rev-parse HEAD` if missing
- The file is in `.gitignore` to avoid merge conflicts while still being included in PyPI releases

## RC Version Format

- **Git tags:** Use dash format (e.g., `v0.6.0-rc0`, `v0.6.0-rc1`)
- **pyproject.toml:** Use PEP 440 format without dash (e.g., `0.6.0rc0`, `0.6.0rc1`)
- PyPI automatically recognizes versions with `rc` suffix as pre-releases
- Users must use `--pre` flag to install RC versions: `uv tool install --pre cicada-mcp`

## Branch Requirements

- **RC releases (tags with `-rc`):** Can be tagged from `dev` or `main` branch
- **Stable releases (no `-rc`):** Must be tagged from `main` branch only
- The GitHub Actions workflow enforces these requirements automatically
